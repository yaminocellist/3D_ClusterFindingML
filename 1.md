  What ntp_info is

  The header literally says event info. It has only 100 rows (= 100 events in
  this file) and contains exclusively per-event globals: event, run, seg, job,
  all the occupancy variables (occ11/116/21/216/31/316), trigger/luminosity
  counters (rawzdc, livezdc, scaledzdc, rawmbd, …, bco, bcotr), and the
  multiplicity summary (ntrk, ntpcseed, nsiseed, nhitmvtx, nhitintt, nhittpot, 
  nhittpcall/in/mid/out, nclusall, nclustpc/pos/neg, nclusintt, nclusmaps, 
  nclusmms).

  No spatial coordinates, no hitID, no per-cluster information — it's the
  "header" for each event. You'll notice the same ~50 metadata branches are
  denormalized into every row of the other three trees (every hit/cluster
  carries its event's occupancy and trigger counts).

  What ntp_cluster is

  Header: svtxcluster => max truth. This is every cluster produced by the 
  seed_island clustering pass before track-finding looks at them. Per-cluster
  branches that don't exist in ntp_hit:

  - Geometry: locx, locy (local in pad frame), plus global x,y,z,r,phi,eta,theta
  - Pad-cell address: phibin, tbin, fee, chan, sampa, plus the cluster centroid
  in phielem, zelem
  - Uncertainties: ex, ey, ez, ephi, and the pull versions pez, pephi
  - Energy: e, adc, maxadc
  - Cluster shape: size, phisize, zsize, pedge, redge, ovlp
  - Drift/calibration: thick, afac, bfac, dcal
  - Linkage hints: trackID, niter

  So ntp_cluster is the bridge layer between raw hits and tracks: each cluster
  aggregates several ntp_hit rows (size ≈ number of constituent hits), and
  trackID flags whether the track-finder picked it up.

  How they relate

  ntp_info  (1 row / event, 100 events)
      │
      ├── ntp_hit       (27.5 M raw pad-time samples — input to clustering)
      │
      ├── ntp_cluster   (3.1 M seed_island clusters — output of clustering)
      │
      └── ntp_clus_trk  (221 k clusters — only those associated to a fitted
  track)

  - ntp_hit → ntp_cluster: many hits per cluster. Your existing
  Cluster+Trk+Info_Check.ipynb already uses the bridge keys (event, layer, tbin,
   phielem, zelem) to match a hit to the cluster cell it belongs to. Cluster
  size/phisize/zsize quantify how many hits were merged.
  - ntp_cluster → ntp_clus_trk: ~7% of clusters survive. ntp_clus_trk has all 
  the same cluster branches plus track-level branches that don't exist in
  ntp_cluster: alpha, beta, resphio/resphi/resz (residuals), seedID, siter, and
  the full helix fit spt, sptot, seta, sphi, syxint, srzint, sxyslope, srzslope,
   sX0, sY0, sdZ0, sR0, scharge, sdedx, spidedx, skdedx, sprdedx, sn1pix, snsil,
   sntpc, snhits.
  - ntp_info → all others: every row in the other three trees carries the
  event-level summary fields (occ*, nhittpc*, nclus*, ZDC/MBD counters).
  ntp_info is just the deduplicated version — useful when you want one row per
  event without scanning 27 M hits.

  In short: ntp_info is the per-event header, ntp_hit is the input to
  seed_island, ntp_cluster is its output, and ntp_clus_trk is the post-tracking
  filtered subset with helix-fit columns attached.

⏺ Why only ~7% survives

  The number itself isn't suspicious — it's exactly what you'd expect for real
  (not simulated) sPHENIX TPC data in a high-occupancy run like 79507. The
  seed_island pass is aggressive and prompiscuous; track-finding is the filter.
  Cluster gets dropped between the two stages when any of these is true:

  - It isn't on any track. Loopers / low-pT spirals never reach the silicon and
  get rejected. Cosmics and stray ionization too.
  - It's noise / distortion artifact. Ion-back-flow, sector-boundary
  distortions, dead pad regions, out-of-time pile-up — seed_island happily forms
   clusters out of these; the helix fit doesn't.
  - It fails the propagation window. Even on a good track, a cluster outside the
   χ²/residual gate (resphi, resz) gets unassigned.
  - It fails silicon matching. TPC seeds that can't be matched to MVTX+INTT hits
   get killed; their TPC clusters drop too.
  - It's geometrically redundant. A track only uses one cluster per layer; if
  seed_island made two on the same pad row near each other, the second is
  discarded.

  Quick sanity check from your ntp_info: ~31k clusters/event survive
  seed_island, ~2.2k end up on tracks. With ntrk typically O(50–200) and ~30–40
  used clusters per TPC track, the on-track count is consistent. The other 93%
  is mostly noise + loopers + duplicates, not "missed by the tracker."

  Which to label first

  Start from ntp_clus_trk, but don't stop there. Here's the reasoning:

  1. It's the only "truth" you actually have. This is real data, so there's no
  MC-particle ground truth. The closest thing to a verified cluster is "a helix 
  fit was willing to use it with reasonable residuals" — that's exactly the
  ntp_clus_trk selection. Every row carries resphi, resz, srzslope, spt etc.
  that let you grade quality before you even open it by hand.
  2. It's the only set that's hand-labelable. 221k vs 3.1M is the difference
  between "feasible with a small team" and "give up." Even within 221k you'll
  want to subsample by event/region.
  3. But it has a known bias. Training only on on-track clusters will teach the
  model "a cluster looks like something on a high-pT, silicon-matched, central
  track." It will not learn to:
    - reject sector-edge distortion blobs (those never make it into
  ntp_clus_trk),
    - find clusters on low-pT loopers (also absent),
    - handle the merge/split ambiguities in dense jets (where one of two
  overlapping clusters often gets dropped from ntp_clus_trk).

⏺ Here's my take, organized by what the metric buys you for hand-labeling rather
   than by formula.

  Tier 1 — flag the bad/merged clusters

  These are the metrics that earn their keep because they catch the things you'd
   otherwise label wrong.

  Metric: Local-max count in (phibin, tbin) image
  What it tells you: Multi-peak ⇒ merged clusters from two close tracks. A
    single-particle cluster has exactly 1 local max.
  How to compute: scipy.ndimage.maximum_filter over an n×n window (3×3 is fine
    for sPHENIX pad pitch), then count pixels equal to the local max AND above
    e.g. 30 % of maxadc.
  ────────────────────────────────────────
  Metric: ADC concentration: maxadc / adc_sum
  What it tells you: Sharp spike (≈1) suggests HV breakdown or noise hit, not
    real diffusion. Real clusters sit in ~0.1–0.4.
  How to compute: Already have both branches in ntp_cluster. Free metric.
  ────────────────────────────────────────
  Metric: Gaussian fit χ²/dof
  What it tells you: High χ² = shape doesn't match an ionization cluster
    (electronics ringing, merge, sparking).
  How to compute: Fit A·exp(-½ Σ⁻¹) to the 2D image; report residual sum /
    (N_pixels - 5).
  ────────────────────────────────────────
  Metric: size vs. predicted-from-angle
  What it tells you: Cluster too small/large for its track angle ⇒ split or
    merged.
  How to compute: See Tier 3 templates below.

  Tier 2 — richer shape (model input, not label)

  Tier 3 — cross-event templates (where the real win is)

  You're right to think across events. The single-cluster metrics above are only
   meaningful relative to what a "normal" cluster looks like for that geometry.
  The trick is to bin by the variables that control shape:

  - layer (drift distance differs across the 48 TPC layers ⇒ diffusion varies)
  - alpha (φ-direction track angle from ntp_clus_trk) — stretches φ
  - beta (z-direction track angle) — stretches z
  - scaledmbd (centrality) — high occupancy ⇒ more merges

  Algorithm for the templates:

  1. Bin every ntp_clus_trk row into (layer, alpha_bin, beta_bin, 
  centrality_bin). Roughly 48 × 10 × 10 × 4 ≈ 19k bins; with 221k clusters you
  have ~10 clusters/bin which is workable.
  2. For each bin, store the empirical mean and 5/50/95 percentile of σ_phi,
  σ_z, maxadc, adc_sum.
  3. For any cluster, compute its pulls: (σ_phi − μ_bin) / std_bin etc. A
  cluster with |pull| > 3 in any of the four dimensions is a candidate for
  "weird → label by hand."

  This converts "label all 221k by hand" into "label the ~1–5 % outliers and use
   the bin templates as silent labels for the rest." That's how you make ground
  truth at scale without doing it cluster-by-cluster.

  One concrete algorithm to glue it together

  Per cluster c in ntp_clus_trk:

  # 1. Bbox in pad/time space from existing branches
  phi_lo = c.phielem - c.phisize          # phielem already = charge-weighted 
  center
  phi_hi = c.phielem + c.phisize
  t_lo   = c.tbin    - c.zsize
  t_hi   = c.tbin    + c.zsize

  # 2. Pull constituent hits (one-time event-keyed groupby on ntp_hit)
  hits = hit_index[(c.event, c.layer)]          # pre-grouped lookup
  mask = ((hits.phibin >= phi_lo) & (hits.phibin <= phi_hi)
        & (hits.tbin   >= t_lo)   & (hits.tbin   <= t_hi)
        & (hits.adc    >  0))
  H = hits[mask][["phibin","tbin","adc"]].to_numpy()

  # 3. Build 2D image (sparse → dense)
  img = np.zeros((int(2*c.phisize+1), int(2*c.zsize+1)), dtype=np.int32)
  img[H[:,0]-phi_lo, H[:,1]-t_lo] = H[:,2]

  # 4. Shape moments (charge-weighted)
  W   = img.sum()
  ii, jj = np.indices(img.shape)
  mu_p  = (img*ii).sum() / W
  mu_z  = (img*jj).sum() / W
  s_p2  = (img*(ii-mu_p)**2).sum() / W
  s_z2  = (img*(jj-mu_z)**2).sum() / W
  rho   = (img*(ii-mu_p)*(jj-mu_z)).sum() / (W*np.sqrt(s_p2*s_z2))

  # 5. Local-max count
  from scipy.ndimage import maximum_filter
  peaks = (img == maximum_filter(img, size=3)) & (img > 0.3*img.max())
  n_peaks = peaks.sum()

  # 6. Gaussian fit residual (optional — closed-form on log-image
  #    or scipy.optimize.curve_fit for ~10 ms/cluster)

  Then append the bin template lookup as the last step: (layer, α_bin, β_bin, 
  cent_bin) → (μ, σ) and store the per-cluster pulls. That's your full
  per-cluster feature row.

  Where to start

  If I had to pick three to implement first and not regret:

  1. Local-max count + ADC concentration — these alone catch most of the
  merged/noise outliers.
  2. Charge-weighted σ_phi, σ_z, ρ — these become the per-cluster model inputs
  you'd train on.
  3. Per-(layer, α_bin, β_bin) template means + 3σ envelope — converts "anomaly"
   from a guess into a quantitative pull.

  The 2D Gaussian fit is nice but expensive at 221k clusters; do it only on the
  subset where the cheaper metrics flag something interesting. Same for entropy
  / skewness — they're refinements once the basics work.
